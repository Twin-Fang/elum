# 모션 시스템 — 이룸(ELUM)

> 애니메이션의 **duration·curve·반응 방식**을 정의한다.
> 색·타이포·간격은 [design-system.md](./design-system.md), 화면 명세는 각 flow 문서를 본다.

출처: [RomRom-FE #689 전체 애니메이션 개선](https://github.com/TEAM-ROMROM/RomRom-FE/issues/689)에서
정립한 토스 모션 철학을 이룸에 맞게 옮겼다.

## 왜 이 문서가 필요한가

디자인 시스템이 애니메이션을 다루지 않으면 **각자 알아서 하거나(제각각) 아예 안 쓴다.**
RomRom은 이걸 방치했다가 나중에 한꺼번에 고쳤다.

```
RomRom 사고 전:  Duration 14단계 분산 · Curve 6종 혼용 · 공통 눌림 위젯 없음
```

**이룸은 아직 그 상태가 아니다.** 현재 duration은 3곳(200ms)뿐이고 `Curves`는 한 번도
쓰이지 않았다. 지금 규칙을 정하면 나중에 15개 파일을 일괄 수정할 일이 없다.

이 문서는 그래서 **사후 정리가 아니라 사전 합의**다.

## 핵심 철학 — 모션은 장식이 아니라 설계

토스의 원칙을 이룸의 맥락으로 옮기면 다음과 같다.

| # | 원칙 | 이룸에서의 의미 |
| --- | --- | --- |
| 1 | **이해를 돕는 움직임** | 목적은 "예쁨"이 아니라 **상태 변화를 알아차리게** 하는 것 |
| 2 | **익숙한 패턴 우선** | 새롭고 멋진 인터랙션보다 **바로 알아보는** 인터페이스 |
| 3 | **플로우 단위 통일** | 화면 하나가 아니라 **동선 전체의 리듬**을 맞춘다 |
| 4 | **시스템화** | duration·curve를 토큰으로 관리한다. 화면에서 숫자를 쓰지 않는다 |
| 5 | **강한 인터랙션은 선택적** | 화려한 모션은 **브랜드 핵심 순간에만** |

> 토스 실패 사례: 서랍형 달력을 만들었는데 사용자가 **존재 자체를 인지하지 못했다.**
> 익숙한 탭 UI로 바꾸자 진입이 크게 늘었다.
> **"멋진 인터랙션" < "바로 알아보는 인터페이스"**

### 이룸에만 있는 제약 — 아동 모드

이룸은 발달장애 아동이 직접 조작한다. 위 원칙에 **두 가지가 더 붙는다.**

| 제약 | 이유 |
| --- | --- |
| 아동 화면 애니메이션은 **300ms 이상** | 급격한 전환은 인지 부하가 크다 |
| 실패·오류에 **빨강·경고 아이콘 금지** | 흔들림(shake)·경고색 모션도 같은 이유로 쓰지 않는다 |
| **접근성 설정을 존중한다** | 움직임에 민감한 사용자가 있다 (아래 §접근성) |

보호자 모드는 일반 성인 기준이므로 표준 토큰을 그대로 쓴다.

## 모션 토큰 (AppMotion)

`lib/core/theme/app_motion.dart` 한 곳에서 관리한다. **화면에서 숫자를 직접 쓰지 않는다.**

### Duration — 5단계

| 토큰 | 값 | 용도 |
| --- | --- | --- |
| `instant` | 100ms | 터치 피드백 |
| `fast` | 200ms | 기본 UI 전환, 선택 상태 변경 |
| `normal` | 300ms | 카드·리스트 등장, 상태 변경 |
| `slow` | 400ms | 페이지 전환, 모달 등장 |
| `emphasis` | 500ms | 온보딩, 특수 진입 |

업계 기준(Material·NN/g)은 **100~300ms**, 복잡한 움직임도 500ms를 넘기지 않는다.
"눈에 잡힐 만큼 길되, 흐름을 방해하지 않을 만큼 짧게."

> **800ms는 너무 느리다.** RomRom이 페이지 전환에 800ms를 쓰다 400ms로 줄였다.

### Curve — 4종

| 토큰 | 값 | 용도 |
| --- | --- | --- |
| `standard` | `easeInOut` | 일반 상태 변화 |
| `entry` | `easeOut` | 등장·진입 |
| `decelerate` | `easeOutCubic` | 감속 |
| `springOut` | `ElasticOutCurve(0.5)` | 버튼 눌림 복귀 |

### stagger

| 토큰 | 값 | 용도 |
| --- | --- | --- |
| `staggerDelayMs` | 30ms | 리스트 아이템이 순차 등장하는 간격 |

리스트가 **동시에 팍 뜨지 않게** 한다. `index * staggerDelayMs`로 지연을 준다.

> 아이템이 20개를 넘으면 마지막 아이템이 600ms 뒤에 뜬다. 긴 목록은 **화면에 보이는
> 앞쪽 몇 개까지만** stagger를 걸고 나머지는 즉시 표시한다.

## 공통 위젯

### AppPressable — 눌림 반응

모든 버튼·카드·아이콘의 터치 반응을 통일한다. **버튼마다 반응이 다르면 안 된다.**

```
누름  → 즉각 scale down (애니메이션 없이 setState — 토스는 "즉각 반응"이 핵심)
뗌    → SpringSimulation 물리 복귀 (살짝 bounce)
```

누를 때 애니메이션을 넣으면 **반응이 굼떠 보인다.** 즉시 줄이고, 복귀만 물리적으로 처리한다.

| 상수 | 값 | 적용 대상 |
| --- | --- | --- |
| `scaleButton` | 0.97 | 버튼 (CTA, FAB 등) |
| `scaleCard` | 0.985 | 카드·리스트 아이템 (면적이 넓다) |
| `scaleIcon` | 0.93 | 아이콘 버튼 (뒤로가기 등 작은 영역) |

면적이 넓을수록 **덜 줄인다.** 큰 카드를 0.93으로 줄이면 과장돼 보인다.

**Spring 파라미터**

```
damping ratio = damping / (2 × √(mass × stiffness))
```

1.0 미만이면 bounce가 생긴다.

- 버튼·카드: `stiffness: 300, damping: 20` → ratio ≈ 0.65 (살짝 bounce)
- 아이콘: `stiffness: 300, damping: 30` → ratio ≈ 0.87 (안정적)

**적용하지 않는 대상**

- 토글·슬라이더 — 드래그로 상태를 바꾸는 위젯이라 누름 scale이 어색하다
- PIN 키패드 — OS 시스템 키패드를 쓰므로 관여하지 않는다

**카드 안에 별도 버튼이 있을 때**

카드 전체가 `AppPressable`인데 안에 메뉴 버튼이 있으면, 메뉴를 눌러도 **카드가 같이 줄어든다.**
버튼을 `AppPressable` 바깥 형제로 빼거나 `Stack > Positioned`로 분리한다.

### AppFadeSlideIn — 등장

opacity 0→1 + 아래에서 살짝 올라오기. `delay`로 stagger를 만든다.
기본값은 `normal`(300ms) + `entry` curve.

### AppSkeleton — 로딩 → 콘텐츠

`AnimatedSwitcher` 기반으로 `fast`(200ms) fade 전환한다.

**빈 화면이나 스피너만 띄우지 않는다.** 로딩 상태는 실제 콘텐츠의 형태를 미리 보여준다.

> 단, 화면 구조가 복잡해 스켈레톤이 오히려 혼란스러우면 스피너가 낫다.
> RomRom도 홈탭은 레이어드 레이아웃이라 스켈레톤을 포기하고 스피너로 되돌렸다.

## 페이지 전환 규칙

| 상황 | 전환 | Duration |
| --- | --- | --- |
| 온보딩 단계 진행 | 수평 슬라이드 (SharedAxis) | `slow` |
| 상세 화면 진입 | 수평 슬라이드 | `slow` |
| 탭 전환 | fade | `normal` |
| 모달·바텀시트 | 아래에서 위로 + fade | `slow` |
| 시작 화면 → 홈 | fade | `slow` |

**뒤로가기는 진입의 역방향이어야 한다.** 들어올 때 오른쪽에서 왔으면 나갈 때 오른쪽으로 간다.

**전환 없는 즉시 교체를 쓰지 않는다.** 화면이 툭 바뀌면 그것이 딱딱함의 원인이다.

## 접근성 — 애니메이션 끄기 ⚠️

OS 접근성 설정에서 "동작 줄이기"를 켠 사용자가 있다. 움직임이 어지럼증을 유발하는
경우가 실제로 있으며, **아동 모드에서는 더 중요하다.**

```dart
// 반복·배경 애니메이션은 이 설정을 반드시 확인한다
final reduceMotion = MediaQuery.disableAnimationsOf(context);
```

- **반복 애니메이션**(배경 그라데이션 이동 등)은 정지하고 정적 상태로 보여준다
- 컨트롤러를 만들지 않거나 `repeat()`을 걸지 않는다 — 보이지 않아도 배터리를 쓴다
- 일회성 전환은 유지해도 되지만, 길면 짧게 한다

## 성능

| 규칙 | 이유 |
| --- | --- |
| 반복 애니메이션은 `RepaintBoundary`로 격리 | 배경이 움직일 때 텍스트까지 다시 그리면 낭비다 |
| `dispose()`에서 컨트롤러를 반드시 정리 | 화면을 벗어나도 계속 돌면 배터리를 먹는다 |
| blur는 비싸다 | 큰 영역에 여러 겹 쌓지 않는다 |
| 주기를 서로 배수로 두지 않는다 | 여러 요소가 같은 자리로 돌아와 **패턴이 눈에 보인다** |

마지막 항목이 실수하기 쉽다. 20·40초로 잡으면 40초마다 정확히 겹친다. **22·28·34초**처럼
서로 나누어떨어지지 않게 잡는다.

## 금지

- `Duration(milliseconds: 250)` 처럼 **화면에서 숫자 직접 쓰기** → `AppMotion` 토큰
- `Curves.easeInOutCubicEmphasized` 같은 **토큰에 없는 curve** 임의 사용
- 전환 없는 **즉시 화면 교체**
- 아동 화면에서 **300ms 미만** 전환
- 실패·오류에 **흔들림(shake)·빨강 점멸**
- 접근성 설정을 무시한 **무한 반복 애니메이션**

> Timer·debounce 같은 **애니메이션이 아닌 Duration**은 토큰으로 바꾸지 않는다.
> `SetupDoneScreen.holdDuration`(1600ms)이 그 예다 — 화면 체류 시간이지 모션이 아니다.
