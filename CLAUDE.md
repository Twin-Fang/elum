# ELUM 저장소 공통 규칙

이 파일은 **저장소 전역 규칙**만 다룬다.
모듈별 기술 스택·코딩 규칙은 각 모듈 문서를 따른다.

- 클라이언트: [client/CLAUDE.md](./client/CLAUDE.md)
- 서버: [server/CLAUDE.md](./server/CLAUDE.md)

## 저장소 구조 (모노레포)

```
elum/
├── client/   # Flutter 앱 (보호자 모드 + 아동 모드)
├── server/   # Spring Boot 서비스 서버 (AI DLP Gateway 포함)
└── docs/     # 기획 · 설계 문서
```

## 작업 프로세스 (모든 작업 공통)

모든 작업은 아래 순서를 따른다.

```
설계 → 이슈 작성 → 구현 → 검증 → 커밋 → 푸시 → 배포 → 완료 보고서 → 라벨 완료처리
```

| 단계 | 내용 | 도구 |
| --- | --- | --- |
| 1. 설계 | 요구사항·접근법을 정리하고 승인받는다. 결과는 `docs/superpowers/specs/`에 남긴다 | `/brainstorming` |
| 2. 이슈 작성 | 아래 제목 규칙에 맞춰 이슈를 등록한다 | `/pro-github` |
| 3. 구현 | `develop`에서 작업한다. **`main`에 직접 커밋하지 않는다** | `/pro-init-worktree` |
| 4. 검증 | 빌드·테스트·동작 확인. **통과를 확인하기 전에 완료라고 말하지 않는다** | `flutter test` / `./gradlew test` |
| 5. 커밋 | 이슈 번호를 연결한다 | `/pro-commit` |
| 6. 푸시 | **사용자가 명시적으로 요청할 때만** | |
| 7. 배포 | develop → main 릴리스 PR을 올린다 | `/pro-changelog-deploy` |
| 8. 완료 보고서 | **이슈가 마무리되면 해당 이슈에 구현 내용을 보고서로 남긴다** | `/pro-report` |
| 9. 라벨 완료처리 | 라벨을 `작업완료`로 바꾼다. **이슈는 닫지 않는다** | `/pro-github` |

**8단계는 생략하지 않는다.** 작업이 끝났는데 이슈에 아무 기록이 없으면 무엇을 왜 그렇게
구현했는지가 커밋 메시지에만 남는다.

### 이슈는 닫지 않는다 (9단계)

작업이 끝나면 **라벨만 `작업완료`로 바꾸고 이슈는 열어둔다.**

```
작업전 → 작업중 → 작업완료   (이슈 상태는 계속 open)
```

닫지 않는 이유 — 닫힌 이슈는 기본 목록에서 사라져 나중에 "이거 왜 이렇게 했지"를
찾을 때 검색이 한 단계 더 필요하다. 라벨로 진행 상태를 표현하면 목록에서 바로 보인다.

**이슈를 닫는 경우는 두 가지뿐이다.**

- 사용자가 명시적으로 닫아달라고 할 때
- 작업이 취소·중단됐을 때 (라벨 `취소`와 함께)

> 라벨 교체는 `set-labels`(전체 교체)를 쓴다. `add-labels`로 `작업완료`만 붙이면
> `작업전`이 남아 상태가 두 개가 된다.

**설계 단계 생략 기준** — 오타 수정, 상수값 변경처럼 접근법을 고민할 여지가 없는 작업은
1단계를 건너뛴다. 판단이 서지 않으면 생략하지 않는다.

## 브랜치 전략 · 배포

```
develop  ← 개발 브랜치 (모든 구현은 여기서)
  ↓ 릴리스 PR
main     ← 배포 브랜치 (default) · 버전·릴리스 노트 자동 관리
```

**배포는 `/pro-changelog-deploy` 하나로 처리한다.**

이 스킬이 develop push → 릴리스 PR 생성 → 릴리스 노트 작성 → automerge까지 진행한다.
직접 PR을 만들거나 main에 머지하지 않는다.

> ⚠️ **`main`에 직접 커밋·푸시하지 않는다.** main은 배포 트리거이므로 의도치 않은 배포가 발생한다.

### 여러 에이전트가 동시에 작업한다

이 레포는 **여러 Claude Code 세션이 `develop`에서 동시에 작업**한다. 각 세션은 서로의 존재나
작업 내용을 알지 못한다. 내가 만들지 않은 변경이 언제든 나타날 수 있다는 전제로 움직인다.

| 상황 | 대응 |
|---|---|
| 작업 시작 전 | `git pull --rebase origin develop`로 최신 상태를 먼저 받는다 |
| 커밋 전 | `git status`로 **내가 만들지 않은 변경**이 섞였는지 확인한다 |
| 커밋 | **`/pro-commit`을 사용한다.** 브랜치명에서 이슈 번호를 뽑아 컨벤션에 맞는 메시지를 만들고, 스테이징할 파일도 함께 확인시켜 준다 |
| 스테이징 | `git add -A`보다 **내가 건드린 경로만** 명시해 담는다 |
| 브랜치 전환 발견 | 다른 세션이 옮긴 것일 수 있다. 임의로 되돌리지 않고 현재 브랜치를 존중한다 |
| 푸시 거부(non-fast-forward) | 강제 푸시하지 않는다. rebase로 통합한 뒤 다시 푸시한다 |

**금지**

- `git push --force` / `--force-with-lease` — 다른 세션의 커밋이 사라진다
- `git checkout .` · `git reset --hard` — 다른 세션의 미커밋 작업을 날린다
- 내 작업과 무관한 파일을 함께 커밋하는 것 — 원인 추적이 불가능해진다

의도를 모르는 변경이 작업트리에 있으면 **건드리지 말고 그대로 둔다.** 내 변경만 커밋하고
나머지는 그 세션이 처리하도록 남겨두는 것이 안전하다.

## 이슈 제목 규칙

모노레포이므로 카테고리를 **2단계**(모듈 + 주제)로 나눈다.

```
{이모지}[{유형}][{모듈}][{주제}] {설명}
```

| 구분 | 값 |
| --- | --- |
| 유형 | `버그` `기능요청` `기능추가` `기능개선` `디자인` `시험요청` |
| 모듈 | `클라` `서버` `관리자` |
| 주제 | `홈` `온보딩` `카드` `설정` 등 짧은 명사 |

유형별 이모지는 `.github/ISSUE_TEMPLATE/`의 각 템플릿을 따른다.
(`❗버그` / `🔧기능요청` / `⚙️기능추가` / `🚀기능개선` / `🎨디자인` / `🔍시험요청`)

**작성 원칙**

- 카테고리는 **한글**로 쓴다 — `client` ❌ → `클라` ✅, `admin` ❌ → `관리자` ✅
- 모듈은 **축약형**을 쓴다 — `클라이언트` ❌ → `클라` ✅
- 주제에 **"화면"을 붙이지 않는다** — `홈화면` ❌ → `홈` ✅
- 템플릿에 없는 유형(`[작업]` 등)을 임의로 만들지 않는다

**예시**

```
⚙️[기능추가][클라][홈] 홈 진입 시 오늘의 카드 요약 노출
⚙️[기능추가][서버][관리자] daisyUI 개편 및 조회 기능 추가
❗[버그][클라][온보딩] PIN 재확인 화면에서 뒤로가기가 동작하지 않음
```

## 버전 관리

`version.yml`이 버전의 단일 출처다. 워크플로우가 자동으로 증가시킨다.

⚠️ **루트 `README.md`의 `## 최신 버전 :` 라인은 워크플로우가 자동 관리한다. 수동으로 편집하지 않는다.**

`PROJECT-COMMON-README-VERSION-UPDATE.yaml`이 main push마다 이 라인을 갱신한다.
아래 제약을 어기면 워크플로우가 실패한다.

- 마크다운 볼드(`**`) 사용 금지
- 정규표현식 특수문자(`*`, `[`, `]`, `^`, `$`) 사용 금지
- 콜론(`:`) 뒤에 버전이 바로 와야 한다

또한 이 워크플로우는 **루트 `README.md`가 존재해야** 동작한다. (`docs/README.md`는 대상이 아니다)

## 브랜치 전략

`develop`에서 작업을 모아 `main`으로 릴리스 PR을 올린다.
버전·릴리스 노트는 워크플로우가 자동 관리한다.

## 서비스 원칙

기능을 추가·변경할 때 아래를 위반하지 않는지 확인한다. 상세는 [docs/README.md](./docs/README.md).

1. **진단명·장애 유형을 수집하지 않는다.** 개인화는 `supportGoals`로만 한다.
2. **보호자 입력 원문을 LLM에 그대로 보내지 않는다.** DLP에서 마스킹 후 최소 정보만 전달한다.
3. **AI 생성 결과는 보호자 승인 후에만** 아동 화면에 노출한다.
4. **한 카드에는 하나의 행동만** 담는다.
5. **원문을 감사 로그·로컬에 저장하지 않는다.** 탐지 유형·건수만 저장한다.
6. **데모는 어떤 실패 상황에서도 끝까지 진행되어야 한다.** AI 실패 시 fallback 필수.

## 예외 처리 · 엣지케이스

**어떤 오류가 나도 화면이 깨지지 않는다.** 설계 단계에서 실패 경로를 함께 정하고, 구현할 때
정상 흐름만 만들고 끝내지 않는다.

| 상황 | 요구 |
|---|---|
| 네트워크 실패·타임아웃 | 빈 화면·무한 로딩 금지. 재시도 버튼과 함께 상태를 보여준다 |
| 응답이 비었거나 형식이 다름 | 파싱 실패로 앱이 죽지 않게 방어하고 빈 상태 UI를 띄운다 |
| 권한 거부(마이크·알림 등) | 거부해도 앱은 계속 동작한다. 왜 필요한지 안내하고 우회 경로를 준다 |
| 예상 못 한 예외 | 화면 전체가 붉은 에러로 덮이지 않게 감싸고, **에러 코드**를 함께 보여준다 |
| 목록이 0건 | 로딩과 구분되는 빈 상태 화면을 별도로 만든다 |

**에러 코드를 반드시 노출한다.** 사용자에게는 "문제가 발생했습니다"로 충분하지만, 화면 어딘가에
`E-1023` 같은 식별자가 있어야 제보를 받았을 때 어디서 터졌는지 추적할 수 있다.

**금지**

- 예외를 잡아서 그냥 삼키는 것 — 실패가 조용히 묻히면 원인을 찾을 수 없다
- `!` 강제 언랩·인덱스 직접 접근 — null·범위 초과로 화면이 통째로 죽는다
- 정상 경로만 구현하고 "나중에 처리" — 그 나중은 대개 오지 않는다

설계 문서에는 **행복 경로와 실패 경로를 함께** 적는다. 실패 경로가 없는 설계는 미완성이다.
