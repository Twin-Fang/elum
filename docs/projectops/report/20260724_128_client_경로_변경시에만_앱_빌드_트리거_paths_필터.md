# client 경로 변경 시에만 앱 빌드 워크플로우가 트리거되도록 paths 필터 추가

## 개요

main 푸시 시 자동 실행되는 Flutter 앱 빌드·배포 워크플로우 4종(FIREBASE·PLAYSTORE·SELFHOSTED·IOS-TESTFLIGHT)에 `paths` 필터가 없어, `server/**`만 변경된 릴리스에서도 앱이 전부 재빌드·재배포되고 있었다(서버 CICD는 이미 `paths: ['server/**']`로 제한된 비대칭 구조). 4개 워크플로우의 `on.push`에 paths 필터를 추가해 client 소스가 실제로 바뀐 푸시에서만 앱 빌드가 돌게 했다.

## 변경 사항

### 워크플로우 (4개 동일 적용)
- `PROJECT-FLUTTER-ANDROID-FIREBASE-CICD.yaml`
- `PROJECT-FLUTTER-ANDROID-PLAYSTORE-CICD.yaml`
- `PROJECT-FLUTTER-ANDROID-SELFHOSTED-CICD.yaml`
- `PROJECT-FLUTTER-IOS-TESTFLIGHT.yaml`

```yaml
on:
  push:
    branches: ["main"]
    paths: ["client/**", "!client/pubspec.yaml"]
```

## 주요 구현 내용

- **`!client/pubspec.yaml` 제외가 핵심이다.** 릴리스 워크플로우의 버전 확정 커밋이 매번 `client/pubspec.yaml`(버전 라인)을 갱신하므로(예: v1.0.91 커밋 확인), `client/**`만으로는 서버-온리 릴리스에서도 필터가 항상 매칭돼 무력화된다. pubspec.yaml만으로는 트리거하지 않게 제외했다.
- 실제 의존성 변경은 `client/pubspec.lock`이 함께 바뀌므로 정상 트리거된다.
- client 변경 없이 재빌드가 필요한 예외 상황은 기존 `workflow_dispatch` 수동 실행으로 커버한다.

## 검증

- 4개 워크플로우 YAML 파싱 검증 통과.
- 이번 릴리스(서버·워크플로우·문서만 변경, PR #131)의 main 머지 푸시에서 **앱 빌드 4종이 모두 트리거되지 않음**을 Actions에서 실측 확인 — 서버 CICD·버전 관리 워크플로우만 실행됐다.

## 주의사항

- 스토어 심사용 앱 재배포가 client 변경 없이 필요한 경우 해당 워크플로우를 `workflow_dispatch`로 직접 실행해야 한다.
