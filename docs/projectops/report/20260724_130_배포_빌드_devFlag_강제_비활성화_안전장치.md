# 배포 빌드에서 devFlag 강제 비활성화 안전장치 추가

## 개요

배포(main) 워크플로우가 `.env` 생성 직후 devFlag(`ELUM_SHOW_DEV_TOOLS`, `ELUM_SKIP_ONBOARDING`)를 **강제로 true로 주입**하는 스텝(해커톤 한정)을 갖고 있어, prod 배포 앱에 개발자 도구 오버레이와 온보딩 건너뛰기가 켜진 채 출시되고 있었다. 이 스텝을 제거하고 반대로 **항상 false를 강제 주입하는 안전장치**로 교체했다. Secret(`CLIENT_ENV_FILE`)에 devFlag가 true로 들어 있어도 배포 산출물에는 항상 false가 실린다.

## 변경 사항

### 배포 워크플로우 (main 푸시 트리거 4종)
- `PROJECT-FLUTTER-ANDROID-FIREBASE-CICD.yaml`: 기존 `Force enable dev tools and skip onboarding (hackathon)` 스텝 2곳(prepare-build·build-android)을 `Force disable dev flags (prod safety)`로 교체.
- `PROJECT-FLUTTER-IOS-TESTFLIGHT.yaml`: 동일 교체 2곳 — 기존에는 build 잡이 SKIP_ONBOARDING을 주입하지 않는 잡 간 불일치가 있었는데, 두 잡 모두 두 키를 다루도록 통일. (BSD sed 문법 유지)
- `PROJECT-FLUTTER-ANDROID-PLAYSTORE-CICD.yaml`: 강제 주입 스텝이 없던 워크플로우 — `.env` 생성 직후 안전장치 스텝을 2개 잡에 신규 추가.
- `PROJECT-FLUTTER-ANDROID-SELFHOSTED-CICD.yaml`: 동일하게 신규 추가.

```yaml
- name: Force disable dev flags (prod safety)
  run: |
    sed -i '/^ELUM_SHOW_DEV_TOOLS=/d' .env
    sed -i '/^ELUM_SKIP_ONBOARDING=/d' .env
    echo 'ELUM_SHOW_DEV_TOOLS=false' >> .env
    echo 'ELUM_SKIP_ONBOARDING=false' >> .env
```

### 유지한 것
- 수동/테스트 빌드(`PROJECT-FLUTTER-ANDROID-TEST-APK.yaml`, `PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT.yaml`)는 테스터용이므로 현행(강제 활성화) 유지.

## 주요 구현 내용

- "true이면 false로 바꾸는" 조건 분기 대신 **무조건 삭제 후 false append** 방식을 택했다 — 조건 누락 여지가 없고, sed가 매칭할 라인이 없어도 스텝이 실패하지 않는다.
- `.env` 생성이 잡마다 따로 일어나는 구조이므로, 실제 빌드 산출물에 반영되도록 `.env`를 만드는 **모든 잡**에 적용했다.

## 주의사항

- 이 변경으로 **심사자/테스터 배포 빌드에서도 온보딩 건너뛰기가 꺼진다** (이전 커밋 d274ff9가 심사 편의를 위해 켰던 동작의 의도적 롤백). 심사·테스트에서 즉시 진입이 다시 필요하면 TEST 계열 워크플로우 빌드를 쓰거나 별도 데모 계정 방식으로 전환해야 한다.
- 정식 출시 전 Secret의 devFlag 값 자체도 false로 정리해 두는 것을 권장한다(안전장치는 최후 방어선).
