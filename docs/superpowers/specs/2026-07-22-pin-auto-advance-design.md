# 보호자 비밀암호(PIN) 4자리 입력 자동 전환 설계

- 이슈: [Twin-Fang/elum#101](https://github.com/Twin-Fang/elum/issues/101)
- 대상: `client/lib/features/onboarding/presentation/pin_screen.dart`
- 작성: 2026-07-22

## 배경 / 문제

온보딩 PIN 화면은 **1단계(비밀암호 생성) → 2단계(재확인)** 2단계로 동작한다.
현재는 두 단계 모두 4자리를 채운 뒤 하단 **"맞춤 설정하기"** 버튼을 눌러야 진행된다.

이름·목표·캐릭터 등 다른 온보딩 단계는 입력과 동시에 흐르는데, 이 화면만 버튼을
한 번 더 눌러야 해서 흐름이 끊긴다. 자동 전환을 막았던 원래 근거는 "자동 전환은
오타를 고칠 틈을 주지 않는다"였다(코드 주석 `pin_screen.dart:64-65`).

## 결정한 방향 — "자동 전환 + 확정만 버튼" (사용자 승인)

4자리를 채우면 자동으로 다음으로 넘기되, **PIN이 실제로 저장·확정되는 마지막 순간만**
버튼으로 남겨 오타 수정 여지를 확보한다.

| 단계 | 트리거 | 동작 |
| --- | --- | --- |
| 1단계 (생성) | 4자리 도달 | 자동으로 2단계로 전환, 입력 초기화. 버튼 불필요 |
| 2단계 (재확인) · 불일치 | 4자리 도달 | 자동 검증 → 즉시 1단계로 리셋 + 안내 문구 |
| 2단계 (재확인) · 일치 | 4자리 도달 | **자동 전환하지 않음.** "맞춤 설정하기" 버튼만 활성화 |
| 확정 | 버튼 클릭 | PIN 저장 후 `onboardingDone`으로 이동 |

원래 주석이 우려한 "오타 수정 틈"은 **2단계 일치 후의 확정 버튼**이 담당한다.
재입력까지는 매끄럽게 자동으로 넘기고, 저장이라는 비가역 지점만 사용자가 명시적으로 누른다.

## 상태 모델

기존 상태를 거의 그대로 쓴다. 별도의 "확정 대기" 플래그는 **두지 않는다.**

- `_firstEntry` (`String?`): null이면 1단계, 값이 있으면 2단계
- `_current` (`_controller.text`): 현재 입력값
- `_errorMessage` (`String?`): 불일치 안내 문구

버튼 활성 조건은 지금과 동일하게 `_current.length == pinLength && _isConfirmStep`으로
표현할 수 있다. 2단계에서 4자리가 채워지고 **일치**한 상태에서만 4자리가 유지되기
때문이다(불일치면 즉시 리셋되어 `_current`가 비므로 버튼이 자동으로 꺼진다).
1단계에서는 4자리 도달 시 즉시 전환하며 입력을 비우므로 버튼이 활성화될 틈이 없다.

## 자동 전환 로직

현재 `_onChanged`(controller 리스너)가 매 입력마다 호출된다. 여기서 4자리 도달을
감지해 단계별로 분기한다.

```
_onChanged():
  이전 안내(_errorMessage) 있으면 지우고 setState (기존 로직 유지)
  else setState (점 다시 그리기)

  // 4자리 도달 시 자동 분기 — post-frame으로 미뤄 리스너 재진입/입력 경합을 피한다
  if _current.length == pinLength:
    1단계면 → _advanceToConfirm()   (자동)
    2단계면 → _verifyConfirm()      (자동)
```

- `_advanceToConfirm()`: `_firstEntry = _current` 저장 → 입력 초기화 → 포커스 유지
- `_verifyConfirm()`:
  - 일치: **아무것도 자동으로 하지 않는다.** 4자리가 유지되어 CTA가 활성화된다
  - 불일치: `_firstEntry = null`(1단계로) → 입력 초기화 → `_errorMessage` 설정

### 왜 post-frame으로 미루는가

`_controller.clear()`는 다시 `_onChanged`를 부른다. 4자리 감지 직후 동기적으로
`clear()`를 호출하면 리스너가 재진입한다. `WidgetsBinding.addPostFrameCallback`으로
전환을 다음 프레임에 실행하고, 재진입 시 `_current.length < pinLength`라 조건에
걸리지 않으므로 안전하다. (또는 전환 직전 4자리 재확인 가드를 둔다)

## CTA 버튼

- 1단계: 버튼은 비활성(입력이 4자리에 닿는 즉시 전환되므로 실질적으로 눌리지 않음)
- 2단계 일치: 활성화 → 클릭 시 저장·이동
- 라벨은 "맞춤 설정하기" 유지 (Figma CTA)

`_onComplete`는 **최종 저장 전용**으로 축소한다. 단계 전환·검증 로직은
`_advanceToConfirm` / `_verifyConfirm`으로 분리해 각 함수가 한 가지 일만 하게 한다.

## 실패 경로 (서비스 원칙 6)

| 상황 | 동작 |
| --- | --- |
| 1단계 오타 입력 | 그대로 2단계로 넘어가지만, 2단계 불일치로 걸러져 1단계부터 다시 |
| 2단계 불일치 | 즉시 1단계 리셋 + "암호가 서로 달라요. 다시 만들어볼까요?" (경고색·아이콘 없이) |
| 키패드 내려감 | 점(`PinDots`) 탭 시 재포커스 (기존 유지) |
| 리스너 재진입 | post-frame 지연 + 길이 가드로 방어 |
