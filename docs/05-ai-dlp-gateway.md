# 05. ELUM AI DLP Gateway — 입력 보호 계층

> 상태: 초안 (브레인스토밍 정리본) — 방향은 확정에 가까움 (서비스의 최전방 보안 차별점),
> 구현 범위는 24시간 기준으로 러프하게 합의됨

## 정의

> **ELUM AI DLP Gateway**
> 생성형 AI로 데이터가 전달되기 전에 개인정보를 탐지하고, 마스킹·일반화한 뒤
> 카드 생성에 필요한 **최소 정보만** 전달하는 입력 보호 계층.

- "완성형 AI DLP 솔루션을 개발했다"라고 표현하지 않는다 — 과장으로 보일 수 있음. 위 정의대로 **입력 보호 계층**으로 정확히 소개한다.
- 위치: **Spring 서비스 서버 내부 모듈**. LLM과 맞닿는 유일한 지점. (초기 안의 Flask AI 서버는 폐기)
- LLM은 **기 구축된 로컬 LLM**이라 데이터가 외부 서비스로 나가지 않지만, "모델이 알 필요 없는 정보는 프롬프트에도 넣지 않는다"는 **최소 정보 원칙**으로 DLP는 그대로 유지한다.

## 핵심 카피

> **아이를 이해하기 위해, AI가 아이의 개인정보까지 알 필요는 없습니다.**

## 명칭 사용 규칙

| 맥락 | 명칭 |
| --- | --- |
| 개발·아키텍처 | **ELUM AI DLP Gateway** |
| 사용자 화면 | **이룸 AI 안심 보호** (+ 작게 `Powered by ELUM AI DLP Gateway`) 또는 간단히 `🛡 AI DLP 보호 적용` |
| 발표 자료 | **AI DLP 기반 Privacy-by-Design 구조** — 부제: "개인정보를 수집한 뒤 보호하는 것이 아니라, 생성형 AI에 전달되기 전에 필요한 정보만 남깁니다." |

## 처리 파이프라인

```
보호자 입력
  ↓
이룸 서비스 서버 (Spring)
  ↓
AI DLP Gateway (Spring 내부 모듈)
  ├─ 개인정보 패턴 탐지 (전화번호·이메일 등 정형 정보 → 정규식)
  ├─ 아동 호칭 치환 ([아동])
  ├─ 학교·기관·장소 일반화
  ├─ 불필요한 세부정보 제거
  ├─ 프롬프트 인젝션 의심 문구 검사
  └─ 보호된 입력문 생성
  ↓
로컬 LLM (기 구축)
  ↓
JSON 형식 행동 카드 생성
  ↓
출력 스키마 및 위험 문장 검수
  ↓
보호자 검토·수정·승인
  ↓
아동 화면
```

**이중 안전 구조**: 입력 단계 = AI DLP가 개인정보 보호 / 출력 단계 = 보호자가 AI 결과 검토·승인.

## 탐지 방식 (기획 방향)

- **정형 정보** (전화번호, 이메일, 주소): 패턴(정규식) 기반 탐지
- **문맥 식별 정보** (이름, 기관명, 장소명): 개체명 인식(NER) 방식으로 탐지 → `[이름]`, `[기관명]`, `[장소]` 형태로 치환
- LLM에는 **마스킹된 문장 + 카드 생성을 위한 최소 정보만** 전달

## 처리 예시 (데모 시나리오)

### 앱 → 서버 입력

```json
{
  "nickname": "하늘이",
  "supportGoals": ["PREPARE_ITEMS", "PREPARE_NEW_SITUATIONS"],
  "routineInput": "내일 비가 많이 올 예정이야. 하늘이가 오전 8시 20분까지 한별초등학교 정문에 갈 수 있게 준비해야 해."
}
```

### DLP 처리 후 LLM에 전달되는 것

```json
{
  "supportGoals": ["PREPARE_ITEMS", "PREPARE_NEW_SITUATIONS"],
  "routineInput": "내일 비가 많이 올 예정이야. [아동]이 학교에 갈 수 있게 준비해야 해."
}
```

**`nickname: 하늘이`는 LLM에 보내지 않는다.**

### 처리 내역

| 원문 정보 | AI DLP 처리 | 처리 이유 |
| --- | --- | --- |
| 하늘이 | `[아동]`으로 치환 | 아동 식별 가능 정보 |
| 한별초등학교 | `학교`로 일반화 | 구체적인 소속 기관 불필요 |
| 정문 | 제거 | 행동 단계 생성에 불필요 |
| 오전 8시 20분 | 제거 | 현재 카드 생성 목적에 불필요 |

> 행동 카드 생성에는 "비 오는 날 등교"라는 상황만 필요하다.
> 아동 이름이나 구체적인 학교 정보는 LLM이 알 필요가 없다.

### 호칭 복원 전략

LLM 응답이 `[아동]이 여벌 양말을 가방에 넣어요.`라면 앱에서 로컬 데이터로
`하늘이가 여벌 양말을 가방에 넣어요.`로 치환할 수 있다.

**더 단순한 방법 (24시간 구현 추천)**: AI에 이름을 보내지 않고, 생성 문장을 모두
이름 없는 형태("가방에 여벌 양말을 넣어요.")로 만들게 한다.

## 24시간 실제 구현 범위

거대한 보안 엔진이 아니라, 아래 네 가지가 **실제로 동작**하면 충분히 설득력 있다.

| 보호 대상 | 구현 방법 |
| --- | --- |
| 아이 호칭 | 온보딩에서 저장한 `하늘이`를 `[아동]`으로 치환 |
| 학교명 | `○○초등학교`, `○○중학교` 패턴을 `학교`로 치환 |
| 전화번호 | 정규식으로 `[전화번호]` 치환 |
| 이메일 | 정규식으로 `[이메일]` 치환 |

+ 프롬프트 인젝션 의심 문구 점검 (간단한 패턴 검사 수준)

### 확장 아이디어 (해커톤 이후 / 여유 시)

- NER 기반 이름·기관·장소 탐지 고도화
- 프롬프트 인젝션 탐지 고도화
- 처리 이력 대시보드

## 감사 로그 정책 (중요)

**원문 자체를 감사 로그에 저장하지 않는다.** 저장한다면 다음 정보만:

```json
{
  "detectedTypes": ["CHILD_IDENTIFIER", "INSTITUTION"],
  "maskedCount": 2,
  "removedCount": 1,
  "createdAt": "..."
}
```

## UI 표현 (러프)

### 로딩 화면 통합안 (추천 — [03-screens.md](./03-screens.md) 페이지 7)

- 3단계 진행 표시 중 2단계에서 보안을 가장 강조:
  > ✓ **AI DLP로 개인정보를 보호했어요**
  > 이름과 학교 정보는 안전하게 가렸어요.
- 하단 고정 문구: 🔒 원문은 AI에 그대로 전달되지 않아요.
- 상태 문구 표현 후보 (탐지 로직을 나열하지 말고 **무엇을 보호했는지만** 보여줄 것):
  1. "이름과 학교 정보를 안전하게 가렸어요." ← 사용자 화면에 가장 자연스러움 (추천)
  2. "아동 식별 정보와 기관 정보를 보호했어요." (조금 더 기술적)
  3. "개인정보 2건을 안전하게 보호했어요." (수량 강조)

### 보호 내역 바텀시트 (심사 데모용 장치)

상태 문구 옆 작은 `보호 내역` 링크 → 바텀시트:

- 제목: **AI에 전달하기 전에 보호했어요**
- 설명: 행동 카드 생성에 필요하지 않은 식별 정보는 다른 표현으로 바꿨어요.
- 전송 전/후 문장 비교:

```
보호자 입력
하늘이가 한별초등학교에 갈 준비를 해야 해.

AI에 전달된 문장
[아동]이 학교에 갈 준비를 해야 해.
```

일반 사용자는 누르지 않아도 되고, 심사 데모에서만 눌러 보여준다.

### 별도 전체 페이지안 (대안, 미채택 기류)

"AI DLP 입력 보호"를 독립 화면으로 두는 안: 배지(`🛡 이룸 AI DLP`), 메인 문구
"AI에 보내기 전, 개인정보를 보호하고 있어요", 처리 3단계 표시, "2개의 식별 정보와 1개의
불필요한 세부정보를 보호했어요" 완료 문구, `보호된 내용으로 계속하기` 버튼.
→ 사용자가 보안 페이지를 매번 통과해야 하는 부담 때문에 로딩 통합안이 우세.

### 일러스트 방향

- 가운데 행동 카드 2~3장 + 카드 앞의 작은 방패 + 우산·가방·양말 아이콘
- 이름표가 방패를 통과하며 `[아동]`으로 바뀌는 짧은 애니메이션
- 돋보기로 검사하는 보안 솔루션 느낌보다 **방패를 거쳐 카드가 만들어지는 장면**이 이룸과 어울림

## 미확정 사항

- **발표 카피 재조정**: 로컬 LLM 전환으로 "외부 AI에 보내지 않는다" 서사가 "데이터가 서버 밖으로 나가지 않는다 + 모델에도 최소 정보만 준다"의 이중 구조로 더 강해질 수 있음 — 발표 문구 다듬기 필요
- DLP UI: 로딩 통합안 vs 별도 페이지안 최종 결정
- 프롬프트 인젝션 검사 패턴 목록
- 학교/기관 일반화 패턴의 범위 (초·중·고 외 복지관, 병원 등)
- 감사 로그 저장 위치 (PostgreSQL vs 파일) 및 보존 기간
