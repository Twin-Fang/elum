# 07. MVP 범위 · 데모 시나리오 (24시간 해커톤)

> 상태: 초안 (브레인스토밍 정리본)

## 이번 24시간에서 만들지 않을 것

| 제외 기능 | 제외 이유 |
| --- | --- |
| 회원가입·로그인 | 데모 핵심 가치와 무관 |
| 보호자·아동 별도 앱 | 하나의 앱에서 모드 전환 가능 |
| 실제 장애 진단 정보 입력 | 불필요한 민감정보 |
| 여러 아동 관리 | 범위 과다 |
| 일정 캘린더 | 핵심 생성 흐름과 무관 |
| 실시간 날씨 API | 보호자 문장에 날씨 포함 |
| 실시간 이미지 생성 | 장애 가능성·대기 시간 |
| 복잡한 보상 상점 | 별 애니메이션으로 충분 |
| 통계·리포트 | 발표 이후 확장 기능 |
| 완전한 PIN 인증 | 전환 UI만 구현 |

- 이미지는 데모 전 미리 제작한 **에셋 5장**을 사용한다.

## 반드시 동작해야 하는 것 (데모 성립 조건)

1. 온보딩: 호칭 입력 + 도움 목표 선택 → 저장
2. 자연어 일과 입력 → AI DLP 마스킹이 **실제로** 동작 (호칭·학교명·전화번호·이메일 4종)
3. 목표를 반영한 AI 추가 질문 1회
4. 행동 카드 5장 생성 → 보호자 검토·수정·승인
5. 아동 모드: 큰 이미지 + TTS + 체크리스트 + "했어요!" → 별 보상
6. AI/백엔드 실패 시 fallback으로 동일 데모 진행 ([06-api-spec.md](./06-api-spec.md))

## 데모 핵심 장면 (10단계)

```
1. 보호자가 도움 목표를 선택
2. 비 오는 날 등교 일과 입력
3. AI DLP가 아동 이름과 학교명을 탐지
4. 전송 전·후 문장을 나란히 표시 (보호 내역)
5. 최소화된 문장만 로컬 LLM에 전달
6. AI가 목표에 맞춰 준비물을 추가 질문
7. 행동 카드 5개 생성
8. 보호자가 카드 수정·승인
9. 아동이 이미지와 TTS로 행동 수행
10. 별 보상 완료
```

두 개의 와우 포인트를 모두 보여준다:

- 초반: **진단명 없는 맞춤 설정** (도움 목표 → AI 맞춤 설정 완료 화면)
- 중반: **입력문이 AI DLP를 거쳐 실제로 바뀌는 장면** (전/후 비교)

## 2분 30초 발표 시연 순서 (러프)

| 시간 | 시연 내용 | 심사위원에게 전달되는 가치 |
| --- | --- | --- |
| 0:00~0:10 | 시작 화면 | 서비스 목적 |
| 0:10~0:20 | `하늘이` 호칭 입력 | 최소 정보 개인화 |
| 0:20~0:35 | 도움 목표 2개 선택 | 아동 맞춤 설정 |
| 0:35~0:45 | AI 카드 생성 기준 완료 | 진단명 없는 개인화 |
| 0:45~ | 일과 입력 → DLP → 추가 질문 → 카드 생성 → 승인 → 아동 수행 → 보상 | 핵심 플로우 + 보안 |

> 0:45 이후 세부 타임라인은 브레인스토밍에서 확정되지 않음 — 리허설하며 배분.

## 개발 일정 (러프)

- 1일 차: 보호자용 일과 입력 화면과 카드 생성 흐름 우선
- (이후 일정은 브레인스토밍에서 구체화되지 않음)

## 데모 안전 수칙

- 추천 카드 자동 입력을 기본으로 쓰고, 문장 입력을 보여주기 위해 마지막 단어 정도만 직접 수정
- 카드 수정 데모 시 구체적인 주소·학교명은 입력하지 않는다 (개인정보 최소 수집 메시지 유지)
- AI 응답이 빨라도 DLP 로딩 연출을 최소 1~2초 유지 (보안 처리 체감)
- 발표 전 fallback 3단계(파싱 실패/API 실패/백엔드 실패) 각각 실제로 리허설

## 미확정 사항

- 시간대별 팀원 역할 분담
- 발표 자료(슬라이드) 구성
- 0:45 이후 시연 타임라인 상세
