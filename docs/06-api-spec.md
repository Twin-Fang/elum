# 06. API 러프 명세 및 실패 대비 (Fallback)

> 상태: 초안 (브레인스토밍 정리본) — 엔드포인트 구성은 러프하며 구현하면서 확정한다.
> 프론트는 백엔드 완성 전에 아래 JSON 형식의 **고정 Mock 데이터로 전체 플로우를 먼저 구현**한다.

## 엔드포인트 (러프)

| 엔드포인트 | 역할 | 우선순위 |
| --- | --- | --- |
| `POST /api/routines` | 일과 생성 (DLP → AI 카드 생성 파이프라인 진입) | 필수 |
| `PUT /api/routines/{id}` | 보호자가 수정한 카드 반영 | 선택 |
| `POST /api/routines/{id}/steps/{stepId}/complete` | 아동 수행 완료 상태 저장 | 선택 |

- 24시간 데모에서는 **수정과 완료 상태를 프론트 로컬 상태로 처리해도 된다.**
- 추가 질문 → 답변 → 최종 카드 생성의 API 분리 방식(단일 엔드포인트 2단계 호출 vs 별도 엔드포인트)은 미확정.

## 요청/응답 JSON (러프 계약)

상세 스키마와 예시는 [04-ai-card-generation.md](./04-ai-card-generation.md) 참고. 요약:

### 일과 생성 요청 (앱 → 서버)

```json
{
  "nickname": "하늘이",
  "supportGoals": ["PREPARE_ITEMS", "PREPARE_NEW_SITUATIONS"],
  "routineInput": "내일 비가 많이 올 예정이야. 아이가 학교에 갈 수 있게 준비해야 해."
}
```

### AI 추가 질문 응답 (서버 → 앱)

```json
{
  "question": "하늘이가 비 오는 날 평소와 다르게 챙겨야 하는 물건이 있나요?",
  "questionType": "MULTIPLE_CHOICE",
  "options": ["우산", "우비", "장화", "여벌 양말", "작은 수건"]
}
```

### 보호자 선택 결과 (앱 → 서버)

```json
{
  "selectedItems": ["우산", "우비", "여벌 양말"]
}
```

### 최종 카드 응답 (서버 → 앱)

```json
{
  "title": "비 오는 날 학교 가기",
  "steps": [
    {
      "order": 1,
      "title": "학교 갈 옷을 입어요",
      "description": "학교에 갈 옷을 차례대로 입어요.",
      "ttsText": "학교에 갈 옷을 입어요.",
      "checklist": [],
      "imageKey": "get_dressed"
    }
  ]
}
```

(steps 5개 전체 예시는 04 문서 참고)

## AI 실패 대비 Fallback 체인 (필수)

**심사 중에는 어떤 상황에서도 같은 데모가 끝까지 진행되어야 한다.**

```
실제 AI 성공
→ 실제 질문·카드 표시

AI 응답 파싱 실패
→ "비 오는 날" 고정 질문 사용

AI API 연결 실패
→ 고정 카드 5개 반환

백엔드 전체 실패
→ 프론트 Mock JSON 사용
```

- 각 단계의 고정 데이터는 데모 시나리오("비 오는 날 학교 가기")와 동일한 내용으로 준비한다.
- 프론트 Mock JSON은 최종 카드 응답 형식과 동일하게 유지해서, 백엔드 연동 시 교체 비용을 없앤다.

## 프론트 작업 순서 (러프)

1. 고정 Mock JSON 정의 (위 형식 그대로)
2. 공통 컴포넌트 구현 ([03-screens.md](./03-screens.md)의 컴포넌트 표)
3. Mock 기반으로 온보딩 → 카드 수행 → 보상까지 전체 플로우 완성
4. 백엔드/AI 연동 교체 + fallback 연결

## 미확정 사항

- 응답 공통 envelope 여부 (`{data, error}` 래핑 등)
- 추가 질문 단계의 API 설계 (세션 개념 필요 여부)
- 에러 코드 체계
- 인증 없음(MVP) 전제에서 사용자 식별 방식 (디바이스 로컬 저장 가정)
