# 02. 시스템 아키텍처

> 상태: 초안 (브레인스토밍 정리본) — 구성 요소와 역할 분담은 합의됨, 세부 스펙은 러프
>
> **변경 이력**: 초기 안의 Python·Flask AI 서버는 **폐기**. DLP·프롬프트 구성·LLM 호출을
> 모두 Spring 서버 내부에서 처리하고, LLM은 **이미 구축된 로컬 LLM**에 연결한다.

## 전체 구성

```
[Flutter 앱 (client/)]
  보호자 모드: 온보딩, 일과 입력, 카드 검토·승인
  아동 모드: 카드 수행, TTS, 별 보상
        │  REST API
        ▼
[서비스 서버 (server/) — Spring Boot · Java 21]
  ├─ 도메인: 사용자/일과/행동 카드/카드 순서/준비물 체크리스트/수행 완료 상태
  ├─ AI DLP Gateway (내부 모듈): 개인정보 마스킹, 프롬프트 인젝션 점검
  ├─ 프롬프트 구성 → 로컬 LLM 호출 → 응답 형식 검수
  └─ 저장소: PostgreSQL
        │
        ▼
[로컬 LLM]  ← 기 구축 완료. 마스킹된 문장 + 카드 생성용 최소 정보만 전달
```

## 기술 스택

| 영역 | 스택 | 비고 |
| --- | --- | --- |
| 앱 | Flutter · Dart | 보호자용 + 아동용을 **하나의 앱에서 모드 전환** (별도 앱 아님) |
| 서비스 서버 | Spring Boot · Java 21 | REST API, 도메인 관리 + **DLP·LLM 호출까지 전담** |
| LLM | Ollama (팀 보유 GPU 서버, 기 구축) | 접속 정보는 저장소 외부에서 관리 (아래 참고) |
| DB | PostgreSQL | 사용자, 일과, 카드, 순서, 체크리스트, 완료 상태 |
| 음성 안내 | Flutter TTS | 카드별 `ttsText`를 클라이언트에서 재생 |
| 이미지 저장 | AWS S3 | 기본 이미지 에셋 (실시간 생성 없음) |
| 협업·배포 | GitHub, Docker, GitHub Actions, AWS EC2, Figma | |

## 역할 분담 원칙

- **앱(client)**: 화면·플로우 전담. 백엔드 완성 전에 **고정 Mock JSON으로 전체 플로우를 먼저 구현**한다. ([06-api-spec.md](./06-api-spec.md)의 fallback 참고)
- **서비스 서버(server)**: 도메인 데이터 관리 + LLM과 맞닿는 유일한 지점. **개인정보가 DLP 모듈을 거치지 않고 LLM 프롬프트에 들어가지 않게 하는 책임**을 가진다.
  - 내부 모듈 경계(러프): `domain`(일과·카드) / `dlp`(마스킹·검사) / `llm`(프롬프트 구성·호출·응답 검수) — 계층을 나눠 DLP를 우회할 수 없는 구조로.
- **로컬 LLM**: 이미 구축되어 있음. 서버에서 HTTP로 호출하는 외부 프로세스로 취급한다.

## 로컬 LLM 상세 (Ollama, 확인 완료)

> ⚠️ **접속 정보(호스트·포트·계정)는 이 저장소에 절대 기록하지 않는다.**
> Spring에서는 환경변수 또는 gitignore된 설정 파일(`application-secret.yml` 등)로 주입한다.

- 팀 보유 GPU 서버(Windows)에 **Ollama 구동 중** — 동작 확인 완료 (2026-07-17)
- Ollama는 **OpenAI 호환 API**(`/v1/chat/completions`)와 자체 API(`/api/chat`)를 모두 제공 → Spring에서 Spring AI 또는 OpenAI 클라이언트 형식으로 연결 가능
- JSON 강제 출력: Ollama `format: json` / structured output 옵션 사용 가능

### 설치된 모델 중 이룸 후보 (한국어 + JSON 출력 기준)

| 모델 | 특징 | 비고 |
| --- | --- | --- |
| `qwen3.5:4b` | tools·thinking 지원, 262k ctx | **1순위 후보** — 품질/속도 균형 |
| `qwen3.5:9b` | 상위 품질 | VRAM 여유 확인 필요, 응답 느릴 수 있음 |
| `exaone-deep:2.4b` / `7.8b` | 한국어 강점 (LG) | thinking 모델이라 응답 시간 유의 |
| HyperCLOVAX-SEED 1.5B/3B | 한국어 특화 (네이버) | 소형이라 카드 JSON 품질 검증 필요 |
| `gemma4:e2b` / `e4b` | tools·thinking 지원 | 대안 |

→ 실제 선택은 **데모 시나리오 프롬프트로 품질·응답속도 테스트 후 결정** (미확정)

### 네트워크 연결 (확인된 제약)

- LLM 포트는 **외부에 개방되어 있지 않음** (확인 완료 — 보안상 바람직)
- 따라서 EC2에 배포된 Spring이 직접 접근 불가. 연결 방식 결정 필요:
  1. 데모 기간 Spring을 LLM과 같은 네트워크(로컬)에서 실행 ← 가장 단순
  2. SSH 터널/리버스 프록시로 제한적 개방
  3. 해커톤 현장 네트워크 구성에 따라 재검토

## 데이터 흐름 (일과 생성 기준)

```
보호자 입력 (자연어)
  ↓
서비스 서버 (Spring)
  ↓
AI DLP Gateway (Spring 내부 모듈)
  ├─ 개인정보 패턴 탐지 (전화번호·이메일 등)
  ├─ 아동 호칭 치환 → [아동]
  ├─ 학교·기관·장소 일반화
  ├─ 불필요한 세부정보 제거
  ├─ 프롬프트 인젝션 의심 문구 검사
  └─ 보호된 입력문 생성
  ↓
로컬 LLM 호출
  ↓
JSON 형식 행동 카드 생성
  ↓
출력 스키마 및 위험 문장 검수 (Spring)
  ↓
보호자 검토·수정·승인 (앱)
  ↓
아동 화면
```

## 로컬 LLM 사용에 따른 보안 서사 (참고)

- 로컬 LLM이므로 **데이터가 외부 서비스로 나가지 않는다는 것 자체**가 추가 보안 포인트가 된다.
- 그래도 AI DLP Gateway는 유지한다 — "모델이 알 필요 없는 정보는 프롬프트에도 넣지 않는다"는
  **최소 정보 원칙**은 로컬·외부와 무관하게 성립하고, 발표 서사의 핵심이다.
- 발표 카피의 "외부 AI" 표현은 "AI/LLM"으로 조정 필요. → [05-ai-dlp-gateway.md](./05-ai-dlp-gateway.md) 미확정 사항 참고

## 이미지 전략 (24시간 제약)

- **실시간 이미지 생성은 하지 않는다.** (장애 가능성·대기 시간)
- 데모 전 미리 제작한 **이미지 에셋 5장**을 사용하고, 카드의 `imageKey`로 매핑한다.
  - 예: `get_dressed`, `pack_socks`, `wear_raincoat`, `take_umbrella`, `walk_in_rain`

## TTS 전략

- 서버에서 음성 파일을 만들지 않고 **Flutter TTS로 클라이언트에서 재생**한다.
- LLM 응답의 `ttsText` 필드를 그대로 읽는다.
- 아동 호칭(하늘이)은 LLM에 보내지 않으므로, 호칭이 필요한 문장은 **앱에서 로컬 데이터로 치환**하거나 처음부터 이름 없는 문장으로 생성한다. (24시간 구현에는 이름 없는 문장이 더 안전 — [05-ai-dlp-gateway.md](./05-ai-dlp-gateway.md) 참고)

## 미확정 사항

- 모델 최종 선택 (위 후보 중 데모 프롬프트 테스트 후 결정)
- 로컬 LLM 호출 타임아웃·재시도 정책
- Spring ↔ LLM 네트워크 연결 방식 (위 1~3안 중 선택)
- PostgreSQL 스키마 상세 설계 (도메인 목록만 합의: 사용자, 일과, 행동 카드, 카드 순서, 준비물 체크리스트, 수행 완료 상태)
- EC2 배포 구성(단일 인스턴스 여부), Docker 구성 상세, 로컬 LLM과의 네트워크 구성
- 데모에서 수정·완료 상태를 서버에 저장할지 프론트 로컬 상태로만 처리할지 (로컬 처리 허용으로 기울어 있음)
